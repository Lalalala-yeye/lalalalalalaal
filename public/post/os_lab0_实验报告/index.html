OS_lab0_实验报告

2026年3月18日 · Lalalala-yeye

本篇为 OS Lab0 的实验报告，重点记录 Thinking 题作答、关键难点分析与实验体会。

一、实验目标与内容#

Lab0 的核心目标不是“算法难度”，而是熟悉命令行编程环境与工具链：

  • 基础 Linux 命令（文件/目录/权限/查找）
  • Shell 脚本参数与批处理
  • grep / sed / awk 文本处理
  • Makefile 的编译组织方式
  • 将代码与脚本按要求复制到指定目录并完成提交

对应题目主要包括：

  1. palindrome.c 回文数判断
  2. src/Makefile 编译 palindrome
  3. hello_os.sh 提取指定行到新文件
  4. changefile.sh 批量删除与重命名目录
  5. search.sh 输出匹配字符串所在行号
  6. modify.sh 字符串替换 + 两层 Makefile 组织构建

二、Thinking（思考题作答）#

注：以下作答结合本实验中实际涉及的源码与脚本文件进行说明。

Thinking 1：为什么 Lab0 要同时考查 C、Shell、sed/grep/awk、Makefile？#

因为真实系统开发不是“只会写 C 函数”就够了。一次完整开发通常包含：

  • 代码实现：如 palindrome.c、fibo.c
  • 批处理与自动化：如 hello_os.sh、changefile.sh
  • 文本检索与替换：如 search.sh、modify.sh
  • 构建与依赖管理：如 Makefile

Lab0 本质是在建立“工具链思维”：源码 + 脚本 + 构建系统协同工作。

Thinking 2：palindrome.c 中回文判断的关键边界是什么？#

核心思路是将输入数 n 逆序得到 rev，再比较 rev == n（或 rev == temp）。

关键边界：

  1. 输入范围：题目给定 1 <= n <= 10000，可忽略负数与超大整数溢出
  2. 含 0 的数字：例如 101 应判为回文，循环逻辑要正确保留中间 0
  3. 循环终止条件：应使用 while (n != 0) 而不是 while (n % 10 != 0)
    若写成后者，遇到末尾为 0 时会提前退出，逻辑不通用

示意代码（核心片段）：

1int temp = n;
2int rev = 0;
3while (n != 0) {
4    rev = rev * 10 + n % 10;
5    n /= 10;
6}
7if (temp == rev) puts("Y");
8else puts("N");

Thinking 3：search.sh 为什么常用 grep -n | awk -F: 组合？#

目标是“输出匹配字符串所在行号”，而 grep -n 直接输出 行号:整行内容，例如：

139:int a = 0;

因此通过 awk -F: '{print $1}' 按 : 分列后取第一列即可。
对应脚本核心：

1grep -n "$2" "$1" | awk -F: '{print $1}' > "$3"

这体现了 Unix 工具哲学：一条命令做一件事，然后管道组合。

Thinking 4：sed 在 modify.sh 中替换字段时，为什么推荐先不加 -i？#

sed -i 是原地修改，若表达式写错会直接改坏文件。更稳妥流程是：

  1. 先 sed "s/$old/$new/g" file 检查标准输出是否正确
  2. 再加 -i 真正落盘：sed -i "s/$old/$new/g" file

如果希望可回滚，使用：

1sed -i.bak "s/$2/$3/g" "$1"

会保留备份文件 *.bak。

Thinking 5：为什么 Makefile 要分“编译 .o”和“链接可执行文件”两步？#

因为 C 项目通常是多源文件，拆两步有两个好处：

  1. 增量编译：只改了 main.c，理论上只需要重编 main.o
  2. 结构清晰：编译规则和链接规则分离，易维护

本实验中外层/内层 Makefile 的职责就是：

  • 内层（code/Makefile）：把 fibo.c、main.c 编译成 .o
  • 外层（csc/Makefile）：链接生成 fibo，并提供 clean

Thinking 6：Shell 脚本处理中为什么要给变量加双引号？#

例如 "$1"、"$3"。
若不加引号，参数含空格或通配符时会发生词拆分与通配展开，导致脚本行为异常。
实验脚本建议遵循：

1grep -n "$pattern" "$input_file" > "$output_file"

而不是不带引号写法。

三、难点分析（含流程图）#

难点 1：文本处理链路设计（grep/awk/sed）#

很多同学会把题目当成“只背命令”，实际难点是：
先把输入/输出数据格式想清楚，再选工具。

示例：search.sh 的数据流：

flowchart LR
    A[输入文件 $1] --> B[grep -n "$2" "$1"]
    B --> C[输出格式: 行号:内容]
    C --> D[awk -F: '{print $1}']
    D --> E[重定向 > "$3"]
    E --> F[结果文件: 仅行号]

关键点：先确认 grep -n 产物，再决定 awk 列提取。

难点 2：批量目录操作脚本的边界与顺序#

在 changefile.sh 中，需求包含删除和重命名，容易出错在：

  • 条件区间边界（41~70 与 71~100）
  • rm -r 与 rmdir 的区别（非空目录必须 rm -r）
  • 循环体中变量展开写法（file$a / "file$a"）

建议流程：

flowchart TD
    A[for a in 1..100] --> B{a >= 71?}
    B -- 是 --> C[rm -r file$a]
    B -- 否 --> D{a >= 41?}
    D -- 是 --> E[mv file$a newfile$a]
    D -- 否 --> F[不操作]

难点 3：Makefile 的“规则语义”理解#

常见误区是把 Makefile 当成“脚本顺序执行”。
实际上它是“目标-依赖”图，make 会按依赖关系决定执行。

简化依赖图：

flowchart LR
    A[fibo 可执行文件] --> B[code/fibo.o]
    A --> C[code/main.o]
    B --> D[code/fibo.c]
    C --> E[code/main.c]

因此“先有 .o，再链接可执行文件”是依赖决定的，不是人工记顺序。

四、实验体会#

  1. CLI 熟练度比记忆命令更重要
    刚开始会觉得命令很多，但真正高频的只有一小部分，重点是“组合”。

  2. 脚本要先保证正确，再追求简短
    尤其 sed -i、rm -r 这类有副作用的命令，先在标准输出验证很关键。

  3. Makefile 是工程化的起点
    从“手敲 gcc”到“make 自动构建”，对后续多文件项目帮助非常大。

  4. 实验报告建议保留思路而不只贴答案
    记录“为什么这样写”，比只记录“最终命令”更有复习价值。

四、Thinking（思考题作答）#

注：以下按“源码/脚本行为 + 工具机制 + 结果验证”方式作答，保证与实现过程对应。

Thinking 1：为什么 make 可以只写目标与依赖就自动决定是否重编译？#

make 会比较目标文件与依赖文件的时间戳：若依赖更新、目标不存在或规则显式触发，才执行命令。
因此在 Lab0 中，当 palindrome.c 发生修改时，make 会重新编译 palindrome；若源码未变化，make 会直接提示目标是最新状态，从而避免重复编译，提高效率。

Thinking 2：Shell 脚本中位置参数（$1、$2、$3）为什么要配合双引号使用？#

在脚本中，$1 等参数常表示文件名或字符串。若不加双引号，参数会经历词分割与通配符展开，可能在包含空格或特殊字符时被错误拆分。
因此像 "$1"、"$2"、"$3" 的写法更稳健，可避免边界输入导致脚本行为异常。

Thinking 3：grep -n ... | awk -F: '{print $1}' 的本质是什么？#

  • grep -n 负责“匹配 + 输出行号:行内容”。
  • awk -F: 负责“按冒号切分字段并提取第 1 列（行号）”。
  • 重定向 > "$3" 负责将标准输出写入结果文件。

这是一种“单一工具做单一职责”的管道组合思路：先定位，再抽取，再落盘，便于调试与替换（如改成 sed -n "/$2/=" "$1"）。

Thinking 4：为什么替换脚本常见 sed "s/$old/$new/g"，g 有什么意义？#

s/old/new/ 默认每行仅替换第一个匹配；加 g 后变为每行全局替换。
在源码修复类任务中，如果同一行出现多个待替换字段，不加 g 会残留旧标识，可能导致编译失败或逻辑不一致。因此在 Lab0 的批量替换场景中，g 更符合预期。

Thinking 5：两级 Makefile 的价值是什么？#

两级 Makefile 将“子模块编译”和“顶层链接”分离：

  • 子目录 code/Makefile 只关心 .c -> .o。
  • 顶层 Makefile 只关心可执行文件链接与清理策略。

这种分层降低耦合，后续新增源文件时只需局部调整规则，便于维护和扩展。

五、难点分析#

难点 1：命令组合与数据流方向容易写反#

核心难点不是单条命令，而是“谁先处理、谁后处理、结果写到哪里”。
例如 search.sh 需要“先匹配，再抽取行号，最后写文件”，任何一步顺序错误都会导致输出不符合题意。

可用流程图表示：

flowchart LR
    A[输入参数: 文件/关键字/输出文件] --> B[grep -n 匹配行]
    B --> C[awk -F: 提取行号]
    C --> D[重定向写入结果文件]
    D --> E[人工核对与脚本复测]

难点 2：文件操作脚本中的边界条件#

例如删除目录、重命名目录时，需要注意：

  • 删除命令是否支持目录（rm -r）
  • 循环区间是否与题目一致（是否包含端点）
  • 已存在目标名时是否会覆盖或报错

可用思维导图方式梳理检查点：

mindmap
  root((changefile.sh))
    循环范围
      41~70 重命名
      71~100 删除
    命令选择
      mv
      rm -r
    健壮性
      参数引用加引号
      重复执行影响

  • 对我帮助最大的实践是：每完成一步就立即运行并检查输出，而不是一次写完再整体排错。
  • 通过这次实验，对后续 OS 实验中的自动化编译、日志检索、批处理修改有了更强信心。
